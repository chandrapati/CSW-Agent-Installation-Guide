# Kubernetes — DaemonSet via Raw Manifest *(community pattern)*

> **Cisco-documented method is the Agent Script Installer.** See
> [Install Kubernetes or OpenShift Agents](https://www.cisco.com/c/en/us/td/docs/security/workload_security/secure_workload/user-guide/4_0/cisco-secure-workload-user-guide-on-prem-v40/deploy-software-agents.html).
> This page covers a **community / practitioner pattern** for
> air-gapped or no-Helm shops that prefer to manage the install
> as raw YAML. Do not hand-author the manifest from the snippets
> below. Generate it with Cisco's installer first, then manage that
> generated output as YAML.

For air-gapped clusters and no-Helm shops, the safe pattern is:
run the Cisco installer in a controlled environment, export the
created resources from the `tetration` namespace, and then manage
those exported resources in Git after review.

> This repo intentionally does **not** ship a static DaemonSet /
> RBAC manifest. The previous examples invented fields that Cisco
> does not publish. Export the resources generated by the Cisco
> installer and manage that output instead.

---

## Prerequisites

- All items from [`../docs/01-prerequisites.md`](../docs/01-prerequisites.md)
- `kubectl` access with cluster-admin
- The exact image referenced by the Cisco-generated DaemonSet,
  mirrored to your internal registry if cluster nodes cannot pull
  from `CFG-SERVER-IP:443`
- The Cisco-generated Secret / ConfigMap fields. Do not invent
  keys or filenames.

---

## Step 1 — Mirror the image

Same as [01-daemonset-helm.md](./01-daemonset-helm.md) Step 2.
Cluster nodes need a path to the image; for air-gapped clusters
that's the internal registry.

---

## Step 2 — Export the Cisco-generated resources

```bash
kubectl get all,cm,secret,sa,role,rolebinding,clusterrole,clusterrolebinding \
  -n tetration -o yaml > csw-agent-generated.yaml
```

Review the exported YAML, remove cluster-generated fields
(`status`, `resourceVersion`, `uid`, etc.), and keep the generated
Secret / ConfigMap field names intact.

---

## Step 3 — Apply the DaemonSet manifest

If you mirror the image, edit only the image reference copied from
the Cisco-generated DaemonSet:

```yaml
spec:
  template:
    spec:
      containers:
        - name: <container-name-from-generated-daemonset>
          image: registry.internal.example.com/csw/<image>:<tag>
```

Apply:

```bash
kubectl apply -n tetration -f csw-agent-generated.yaml
```

Watch the rollout:

```bash
kubectl -n tetration get ds -w
kubectl -n tetration get pods -o wide
```

Expected: `DESIRED == CURRENT == READY` matches your node count
within 1–2 minutes.

---

## Step 4 — Verify

Same as [05-verification.md](./05-verification.md). The cluster
nodes should appear in the CSW UI within minutes of pods reaching
`Running` state.

---

## Step 5 — Day-2 patching cadence

In a no-Helm world, upgrades are an `image:` field bump:

```bash
# Regenerate with the Cisco installer for the target version, then
# review the diff against your checked-in YAML.

# Apply
kubectl apply -n tetration -f csw-agent-generated.yaml

# Watch the rolling update
kubectl -n tetration rollout status ds/<daemonset-name-from-generated-output>
```

Add a PodDisruptionBudget if your DaemonSet is on nodes where
disruption needs to be controlled:

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: csw-agent
  namespace: tetration
spec:
  maxUnavailable: 1
  selector:
    matchLabels:
      app: <label-from-generated-daemonset>
```

## When this is the right method

- **Air-gapped clusters** where the chart repo isn't reachable
- **Clusters where Helm is not permitted** (some regulated /
  classified environments)
- **GitOps shops that prefer raw YAML over chart rendering** —
  Argo CD and Flux both consume raw manifests cleanly

## When the Helm chart is preferable

- Clusters with internet egress and standard Helm tooling
- Multi-cluster fleets where chart values can templatise
  per-cluster differences (region, scope, image version) cleanly
- Whenever you want Cisco's release-cycle hardening of the chart
  rather than maintaining the manifest yourself
